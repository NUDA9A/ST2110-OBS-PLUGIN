# ST2110-OBS-PLUGIN — Plan rules

> Этот файл содержит рабочие правила, правила проектирования, цели этапов и список стандартов/референсов.
>
> Основные файлы управления проектом:
> - `plan.md` — backlog/status-файл: Done, Spec notes / deviations и task backlog / phases.
> - `plan_rules.md` — рабочие правила, правила проектирования, цели этапов и референсы.
> - `code_map.md` — карта production-кода.
> - `tests_file_map.md` — карта test targets / `.cpp` тестов.
> - `conventions.md` — компактные coding/validation/extensibility conventions.
>
> Источники контекста для этого проекта:
> - подключенный репозиторий — основной источник production/test кода, а также актуального состояния `plan.md`, `code_map.md`, `tests_file_map.md` и остальных файлов проекта, кроме специально закрепленных ниже исключений;
> - Project в ChatGPT — основной источник загруженных PDF стандартов и дополнительных материалов вне репозитория, а также основной источник закрепленных control docs `plan_rules.md` и `conventions.md`;
> - пользовательские сообщения в чате — источник локальных незакоммиченных изменений, временных решений и новых требований, которые еще не попали в репозиторий.

## 1. Рабочий процесс

### 1.1. Общая модель работы
- 1 задача за раз (маленькая, 15–60 мин).
- Пока задача не принята, к следующей не переходим.
- Реализацию пишет пользователь.
- Ассистент не должен писать полную реализацию задачи, если пользователь явно не попросил об этом.
- По умолчанию ассистент дает:
  - архитектурные требования;
  - список точечных изменений;
  - exact scope of changes;
  - expected behavior;
  - готовые тесты.
- Если задача добавляет или меняет production API / header / helper boundary / class / struct / enum / free function / новый production-файл, ассистент по умолчанию также обязан дать готовый к копированию API skeleton.

### 1.2. Что делает ассистент до реализации
- Ассистент должен:
  - сформулировать следующую задачу;
  - подробно описать, что именно нужно сделать;
  - сверить задачу с актуальным архитектурным и стандартным контекстом;
  - сразу написать / предложить тесты под задачу.
- Если задача затрагивает production API, public headers, internal architectural headers, helper boundaries или добавляет новый production-файл, ассистент должен сразу дать готовую API-форму изменения.
- Под “готовой API-формой изменения” понимается:
  - точные сигнатуры;
  - все нужные enum / struct / class / function / helper declarations;
  - нужные include’ы;
  - `namespace st2110`;
  - полный skeleton каждого нового файла;
  - полный replacement relevant file, если изменение в существующем файле существенно и без полного файла легко ошибиться.

### 1.2.A. Формат выдачи API и skeleton-кода
- Если задача меняет существующий API или добавляет новый API, ассистент обязан выдавать API в копируемом виде, а не только в виде списка идей или краткого описания.
- По умолчанию:
  - каждый новый production `.hpp` / `.cpp` файл должен быть прислан целиком;
  - для существующих production `.hpp` файлов ассистент должен присылать только новые / измененные declaration blocks, нужные для задачи, а не полный replacement файла;
  - полный replacement существующего production header допустим только если пользователь явно попросил именно полный файл.
- Для существующего `.hpp` файла ассистент не должен заново присылать:
  - неизмененные объявления;
  - существующие inline implementation bodies;
  - части файла, которые не относятся к текущей задаче.
- Если у существующего метода / helper’а меняется intended behavior, но не меняется сигнатура, ассистент обязан:
  - не печатать заново весь файл;
  - явно описать ожидаемое новое поведение реализации.
- Для каждого нового production-файла ассистент обязан включить:
  - все нужные `#include`;
  - header guard или другой уже принятый в проекте формат защиты заголовка;
  - `namespace st2110`;
  - все требуемые enums / structs / classes / free helpers;
  - точные сигнатуры всех методов и helper-функций.
- Все методы классов и free helper-функции должны быть записаны как объявления, заканчивающиеся `;`, в форме, готовой к тому, чтобы пользователь заменил `;` на `{ ... }` с реализацией.
- Ассистент не должен оставлять “неявные” helper’ы только в prose, если они реально требуются для предложенной структуры API.
- Для каждого метода / helper-функции ассистент обязан отдельно описать ожидаемое поведение реализации.
- Описание ожидаемого поведения должно включать, где применимо:
  - требования к входным данным и validation boundary;
  - success path;
  - state changes;
  - возвращаемые значения / ошибки;
  - invariants после вызова;
  - явные ограничения временного MVP-support, если они есть.
- Это описание должно быть достаточно полным, чтобы пользователь мог реализовать тело метода без угадывания intended semantics.
- Тестовые `.cpp` файлы ассистент по-прежнему обязан присылать целиком.
- Ассистент по умолчанию не должен писать body реализации, если пользователь явно не попросил full implementation.

### 1.3. Что делает ассистент после реализации
- После завершения задачи пользователь присылает код или diff / измененные файлы.
- Ассистент проверяет:
  - соответствие задаче и плану;
  - отсутствие ненужных упрощений;
  - отсутствие расхождений со стандартом;
  - соответствие правилам проектирования;
  - расширяемость архитектуры;
  - качество тестового покрытия под задачу.

### 1.4. Сложные детали реализации
- Сложные детали реализации можно разбирать в отдельном чате.
- Приемка задачи выполняется только по коду и тестам.

## 2. Обязательный контекст перед задачей и приемкой

### 2.1. Базовые источники контекста
- Перед каждой новой задачей, перед каждой приемкой реализации и перед любым утверждением вида “изучил правила / работы / контекст” ассистент обязан опираться на актуальные:
  - `plan.md` из подключенного репозитория;
  - `code_map.md` из подключенного репозитория;
  - `tests_file_map.md` из подключенного репозитория, если задача затрагивает тесты, покрытие, приемку тестов или выбор test file;
  - `plan_rules.md` из активного Project;
  - `conventions.md` из активного Project;
  - релевантные `.hpp` / `.cpp` / test files из репозитория;
  - все предоставленные `.pdf` файлы стандартов ST 2110 / RP 2110, доступные в активном Project на текущий момент.

- Для этого проекта действует жесткое правило:
  - ассистент обязан не просто “учитывать” или “помнить” эти файлы, а **реально прочитать их актуальные версии** перед текущим шагом;
  - нельзя опираться только на память из прошлых сообщений, частичные сниппеты, ранее прочитанные версии или собственные предположения о содержимом файлов.

- Под “реально прочитать” понимается:
  - открыть и прочитать актуальное содержимое обязательных `.md` файлов для текущего шага;
  - открыть и прочитать все загруженные в Project стандарты ST 2110 / RP 2110 в PDF, доступные на текущий момент;
  - использовать результат этого чтения при формулировании следующей задачи, при приемке и при обновлении правил/карт, а не считать это формальной галочкой.

- Для PDF стандартов действует отдельное правило:
  - перед следующей задачей и перед приемкой ассистент обязан прочитать **все** доступные в Project стандарты ST 2110 / RP 2110, а не только тот PDF, который кажется наиболее очевидно релевантным;
  - это нужно для того, чтобы не пропустить cross-document requirements, когда одна задача пересекает системные требования ST 2110-10, essence-specific требования ST 2110-20 / ST 2110-30, timing/compliance из ST 2110-21 и measurement practices из RP 2110-25.

- Для этого проекта считается, что **каждая задача потенциально требует сверки со стандартом**, даже если она выглядит как небольшое архитектурное, тестовое, backend-, CLI- или cleanup-изменение.

- Если какой-либо обязательный `.md` файл или какой-либо загруженный стандартный PDF не был реально прочитан в актуальной версии для текущего шага, ассистент не должен:
  - писать, что “изучил правила / работы / контекст”;
  - утверждать, что стандартный и проектный контекст полностью проверен;
  - переходить к формулированию следующей задачи или к полной приемке как к обоснованным действиям.

- Ассистент не должен переходить к формулированию следующей задачи или к приемке текущей реализации, если в доступных источниках отсутствуют обязательные проектные `.md` файлы или какие-либо обязательные PDF стандарты, уже заявленные как используемые в этом Project.

### 2.2. Порядок использования источников
- Ассистент обязан использовать источники в следующем порядке:
  1. релевантные production/test файлы из подключенного репозитория;
  2. project control docs:
    - `plan.md`, `code_map.md`, `tests_file_map.md` — из подключенного репозитория;
    - `plan_rules.md`, `conventions.md` — из активного Project;
  3. PDF стандарты и дополнительные материалы из Project;
  4. пользовательские локальные изменения / фрагменты, если они новее состояния репозитория.

- Для целей поведения ассистента внутри чата:
  - `plan_rules.md` и `conventions.md`, загруженные в активный Project, считаются authoritative runtime copy;
  - версии этих файлов в репозитории должны поддерживаться синхронными, но при расхождении в рамках текущего чата ассистент должен опираться именно на версии из Project;
  - `plan.md`, `code_map.md`, `tests_file_map.md` authoritative берутся из репозитория, если пользователь явно не прислал более свежие локальные изменения.

- Ассистент **не должен** просить пользователя повторно прислать:
  - `plan_rules.md` и `conventions.md`, если они уже загружены в активный Project;
  - `.md` файлы, уже доступные в репозитории;
  - production/test файлы, уже доступные в репозитории;
  - PDF стандарты, уже загруженные в активный Project.

- Ассистент обязан запросить у пользователя конкретный файл / фрагмент только если:
  - нужный файл отсутствует в текущем доступе;
  - в чате обсуждаются локальные незакоммиченные изменения, которых нет в репозитории;
  - доступна только неполная / устаревшая / недостаточная версия файла;
  - нужный стандартный фрагмент отсутствует в Project или недостаточен для уверенной проверки.

### 2.3. Обязанность проверять полноту контекста
- Перед формулированием задачи и перед приемкой ассистент обязан проверить, хватает ли текущего доступа для:
  - проверки соответствия правилам;
  - проверки соответствия архитектуре;
  - проверки соответствия стандартам;
  - выбора правильных production/test файлов;
  - выявления новых стандартных или архитектурных deviations.

- Эта проверка полноты контекста включает не только наличие файлов, но и факт их реального чтения.
- Наличие файла в репозитории, Project или памяти чата само по себе не считается достаточным.
- Для текущего шага ассистент обязан отдельно проверить, что он действительно прочитал:
  - все обязательные `.md` файлы в их актуальных версиях;
  - все загруженные в Project стандарты ST 2110 / RP 2110 в PDF, доступные на данный момент.

- Ассистент не должен делать вид, что контекст проверен, если он:
  - не читал один из обязательных `.md` файлов;
  - не читал один из обязательных PDF стандартов;
  - читал только устаревшую версию файла;
  - опирался только на пересказ, карту, сниппет или собственную память вместо актуального содержимого.

- Если чего-то не хватает, ассистент обязан явно назвать, **чего именно не хватает**, и запросить только это.
- Если какой-то обязательный `.md` файл или PDF недоступен, ассистент обязан остановиться на уровне частичного ответа и не выдавать следующий task proposal / acceptance как полностью обоснованные.

### 2.4. Работа с локальными незакоммиченными изменениями
- Подключенный репозиторий считается источником истины только для уже доступного в нем состояния.
- Если пользователь менял локальные файлы и эти изменения еще не попали в репозиторий, ассистент обязан опираться на присланные пользователем diff / новые версии файлов как на более свежий источник.
- В таких случаях ассистент не должен делать вид, что состояние репозитория уже содержит эти изменения.

### 2.5. Обязательный запрос `.hpp` / `.cpp` при точных изменениях API
- Если задача подразумевает изменение существующих `.hpp` файлов, ассистент обязан опираться на фактическое содержимое соответствующих `.hpp` из репозитория или из присланных пользователем локальных изменений.
- Исключение допустимо только если:
  - актуальное содержимое нужного `.hpp` уже полностью доступно в текущем контексте;
  - либо задача остается чисто архитектурной и не требует точного API diff.
- `code_map.md` используется для выбора релевантных headers, но не заменяет фактическое содержимое `.hpp`, если задача требует точного изменения API / struct / enum / function signatures.

### 2.6. Запрос дополнительных файлов
- Ассистент может запросить только те файлы, которые относятся к текущей задаче или приемке.
- Обычно это:
  - public headers;
  - связанные `.cpp`;
  - конкретные существующие test files;
  - релевантные `.md`;
  - релевантные `.pdf` / выдержки стандартов.
- Если для формулирования задачи или приемки достаточно информации из репозитория и Project, ассистент не должен запрашивать лишние файлы.

## 3. Правила ведения проектных файлов

### 3.1. `plan.md`
- `plan.md` является backlog/status-файлом.
- `plan.md` хранит:
  - task backlog / phases;
  - `Spec notes / deviations`;
  - при необходимости дополнительные status-разделы, если пользователь их реально использует.
- По умолчанию для этого проекта выполненные задачи отмечаются как `[x]` **по месту их объявления** в backlog / phase sections.
- Ассистент не должен по умолчанию переносить выполненные задачи в `## Done` и не должен предлагать пользователю такое перемещение, если пользователь явно не попросил вести `plan.md` именно через перенос в `## Done`.
- Если в `plan.md` уже существует исторический раздел `## Done`, это не означает, что новые выполненные задачи автоматически должны дублироваться или переноситься туда.

### 3.2. `code_map.md`
- `code_map.md` должен поддерживать описание актуальных production-файлов реализации.
- Для каждого значимого production-файла реализации, прежде всего public headers, в `code_map.md` должны быть указаны:
  - архитектурная роль файла;
  - основные зависимости / связи с другими файлами;
  - перечисление ключевых enum / struct / class / function / method с кратким назначением.
- Если фактический production-код расходится с описанием файла в `code_map.md`, приоритет у фактического кода.
- Такое расхождение должно быть устранено обновлением `code_map.md` после приемки текущей задачи.

### 3.3. `tests_file_map.md`
- `tests_file_map.md` должен поддерживать описание актуальных test targets / `.cpp` тестов.
- Если фактические тесты расходятся с описанием в `tests_file_map.md`, приоритет у фактических тестов.
- Такое расхождение должно быть устранено обновлением `tests_file_map.md`.

### 3.4. `conventions.md`
- `conventions.md` хранит компактные локальные coding/validation/extensibility conventions.
- Он не заменяет `plan_rules.md`, а дополняет его.
- При конфликте между краткой convention и более подробным rule из `plan_rules.md` приоритет у `plan_rules.md`, если только пользователь явно не обновил оба файла согласованно.
- Для целей поведения ассистента в чате актуальная версия `conventions.md` берется из активного Project.
- Версия `conventions.md` в репозитории должна обновляться пользователем так, чтобы оставаться синхронной с версией, закрепленной в Project.

### 3.5. Требования к подробности карт
- Описание в `code_map.md` / `tests_file_map.md` должно быть достаточно подробным, чтобы ассистент мог восстановить архитектурный контекст без повторной пересылки всего кода и тестов.
- Карты не должны копировать полную реализацию.

### 3.6. Обновление файлов после принятой задачи
- Текущая задача не считается полностью завершенной и принятой, пока не выполнены оба шага:
  - реализация и тесты проверены в рамках задачи;
  - все необходимые проектные `.md` файлы обновлены пользователем и затем проверены ассистентом.

- После завершения реализации и прохождения тестов ассистент обязан до перехода к следующей задаче определить, какие проектные `.md` файлы должны быть обновлены:
  - `plan.md`, если изменился статус задач, backlog или `Spec notes / deviations`;
  - `code_map.md`, если добавлены production-файлы или изменились публичные структуры, классы, методы, связи или архитектурная роль production-файлов;
  - `tests_file_map.md`, если добавлены, удалены или существенно изменены тестовые файлы / test targets;
  - `plan_rules.md`, если в процессе работы уточнены рабочие правила;
  - `conventions.md`, если изменились устойчивые coding/validation/extensibility conventions.

- При обновлении `plan.md` по умолчанию статус задачи должен отражаться как отметка `[x]` по месту объявления задачи.
- Ассистент не должен по умолчанию перемещать выполненную задачу в `## Done` и не должен требовать такого перемещения от пользователя.
- Если пользователь использует иной workflow ведения `plan.md`, ассистент может следовать ему только после явного подтверждения этого workflow пользователем или самим содержимым файла.

- Ассистент должен выдать пользователю готовые заменяемые блоки для соответствующих `.md` файлов до перехода к следующей задаче.

- После этого пользователь вносит изменения в `.md` файлы и присылает обновленные блоки / файлы обратно в чат.

- Ассистент обязан проверить, что изменения в `.md` файлах внесены корректно:
  - правильные задачи отмечены как выполненные;
  - не закрыты задачи, которые еще не завершены;
  - статус отражен в том месте `plan.md`, которое соответствует фактическому workflow пользователя;
  - новые production/test файлы отражены в соответствующих картах;
  - описание не расходится с фактической архитектурой и реализацией;
  - новые временные ограничения или deviations не потеряны.

- Только после проверки обновленных `.md` файлов ассистент может считать текущую задачу принятой и переходить к следующей задаче.

### 3.7. Формат обновлений проектных файлов
- Когда ассистент предлагает обновить `plan.md`, `plan_rules.md`, `code_map.md`, `tests_file_map.md`, `conventions.md` или другой `.md` файл проекта, он должен давать заменяемые блоки целиком, а не точечные вставки по отдельным строкам.
- Минимальный предпочтительный формат обновления:
  - целый раздел для `plan_rules.md`;
  - целый task / subsection для `plan.md`;
  - целый file block для `code_map.md`;
  - целый test file block или целый subsection для `tests_file_map.md`;
  - целый section для `conventions.md`, если он затронут.
- Для `plan_rules.md` и `conventions.md` ассистент по умолчанию должен давать полный обновленный файл целиком, если изменения затрагивают их рабочее поведение или приоритеты источников.
- Точечные вставки допустимы только если пользователь явно попросил именно точечный diff.
- Если изменение затрагивает несколько мест, ассистент должен дать каждый заменяемый блок отдельно и явно указать, какой существующий блок заменить.
- Обновляемый блок должен быть самодостаточным: пользователь должен иметь возможность заменить старый блок новым без ручной сборки из нескольких фрагментов.

## 4. Правила тестов

### 4.1. Предпочтение существующим тестовым файлам
- Ассистент не должен добавлять новый тестовый файл, если задачу можно корректно покрыть обновлением существующего тестового файла.
- Для выбора подходящего существующего теста ассистент обязан использовать `tests_file_map.md`.
- Если для корректного обновления существующего теста нужно фактическое содержимое файла, ассистент должен взять его из репозитория или запросить только если он недоступен.
- Новый test target / новый `.cpp` файл допустим только если:
  - нет подходящего существующего тестового файла;
  - новая задача покрывает отдельный subsystem / boundary;
  - добавление в существующий тест сделало бы покрытие хуже структурированным или менее поддерживаемым.

### 4.2. Формат выдачи тестов ассистентом
- Когда задача требует новые тесты или обновление существующих тестов, ассистент обязан присылать:
  - точную новую строку `add_st2110_test(...)` для `tests/CMakeLists.txt`, если новый test target действительно нужен;
  - полный готовый `.cpp` файл каждого нового теста;
  - полный готовый `.cpp` файл для каждого существующего теста, который нужно заменить целиком.
- Ассистент не должен ограничиваться описанием тестовой идеи.
- Все новые или обновленные тестовые файлы должны быть даны целиком в копируемом виде.

### 4.3. Правило о скопированных тестах
- Если пользователь не сообщил об изменениях в тестах, которые ассистент ранее дал целиком, для дальнейшей приемки считается, что они были скопированы без изменений.
- По умолчанию считается, что:
  - все тестовые `.cpp` файлы, которые ассистент прислал в полном виде;
  - и соответствующие строки `add_st2110_test(...)`, которые ассистент дал для `tests/CMakeLists.txt`,
    были пользователем добавлены без изменений, если пользователь явно не сообщил обратное.

### 4.4. Когда пользователь обязан / не обязан присылать тесты
- Для приемки задачи пользователь не обязан пересылать файлы из `tests/`, если:
  - они доступны в репозитории в актуальном виде;
  - или были ранее даны ассистентом целиком и скопированы без изменений.
- При приемке задачи ассистент не должен повторно запрашивать содержимое файлов из `tests/`, кроме случаев, когда:
  - пользователь сам изменил тесты локально и эти изменения еще не доступны;
  - пользователь сообщает о падении / адаптации тестов;
  - для приемки нужно проверить расхождение между ранее выданным тестом и фактическим состоянием репозитория;
  - задача требует обновления существующего теста, содержимое которого недоступно.

## 5. Правила формулирования задач

### 5.1. Обязательная сверка
- Каждая задача должна быть сформулирована только после того, как ассистент реально прочитал:
  - `plan.md`;
  - `plan_rules.md`;
  - `code_map.md`;
  - `tests_file_map.md`, если задача затрагивает тесты;
  - релевантные `Spec notes / deviations`;
  - релевантные `.hpp` / `.cpp` / test files из репозитория;
  - все доступные в Project PDF-файлы стандартов ST 2110 / RP 2110.

- Для этого проекта не допускается формулировать задачу:
  - без чтения актуальных project control `.md` файлов;
  - без чтения всех доступных стандартных PDF;
  - на основании предположения, что “эта задача слишком маленькая и стандарт не нужен”.

- Даже если задача кажется внутренней, технической или небольшой, ассистент обязан проверять:
  - не закрепляет ли она отклонение от стандарта;
  - не ломает ли уже заложенные standards-aware boundaries;
  - не вводит ли hardcoded assumption вместо modeled axis / validation boundary;
  - не конфликтует ли с уже известными ST 2110 requirements;
  - не требует ли обновления `Spec notes / deviations`.

- Ассистент не должен писать или подразумевать, что задача сверена с правилами и стандартами, если это чтение не было выполнено реально для текущего шага.

### 5.2. Архитектурная проверка
- При формулировании задачи ассистент обязан проверять не только текущую функциональность, но и соответствие целевой архитектуре.
- В частности:
  - `PixelFormat` и `VideoScanMode` рассматриваются как независимые оси модели;
  - progressive-only поведение в MVP допустимо только как локализованное ограничение;
  - новые изменения не должны закреплять assumptions вида `marker == end of frame`, `timestamp group == frame`, `F is always zero` вне специально выделенных policy / mode-aware точек.

### 5.3. Запрет на закрепление отклонений
- Каждая задача должна быть сформулирована так, чтобы сама по себе не закрепляла архитектурные или стандартные отклонения.

### 5.4. Новые найденные расхождения
- Если во время формулирования задачи обнаружено новое расхождение со стандартом или правилами, ассистент обязан:
  - явно сообщить об этом;
  - добавить расхождение в `Spec notes / deviations`;
  - при необходимости превратить его в отдельную задачу.
- Если расхождение относится к текущей задаче и может быть исправлено без взрывного роста сложности, задача должна быть сформулирована так, чтобы исправить его сразу, а не откладывать.

- При этом ассистент не должен создавать новый `Spec notes / deviations` пункт, если обнаруженное ограничение:
  - уже известно по текущему `plan.md`;
  - уже покрыто существующей запланированной follow-up задачей;
  - и текущая задача не добавляет нового ухудшения, нового класса риска или нового отклонения сверх уже известного ограничения.
- В такой ситуации ассистент должен:
  - сослаться на уже существующую задачу в backlog;
  - при необходимости уточнить формулировку самой задачи;
  - но не дублировать уже запланированное ограничение как новый deviation.
- Новый deviation должен добавляться только если обнаружено действительно новое расхождение, которое:
  - не отражено в текущем `plan.md`;
  - не покрыто существующей задачей;
  - либо возникло как новый побочный эффект / новый риск именно из-за текущего изменения.

### 5.5. Формат постановки production/API задач
- Если задача меняет production API, public/internal header boundary или helper boundary, ассистент обязан формулировать ее не только через prose, но и через готовую структуру кода.
- В такой постановке ассистент обязан:
  - явно указать, какие файлы меняются;
  - дать полный skeleton каждого нового файла;
  - для существующего production `.hpp` файла дать только те declaration blocks, которые реально меняются или добавляются;
  - перечислить все новые / измененные методы, helper’ы, структуры, enum’ы и классы;
  - отдельно описать expected behavior для каждого нового метода / helper’а;
  - отдельно описать expected behavior уже существующих методов, если меняется их intended semantics.
- Ассистент не должен ограничиваться указаниями вида:
  - “добавь helper”;
  - “введи state”;
  - “сделай метод start/stop”;
    если точная сигнатура и intended semantics уже могут быть определены из контекста задачи.
- Ассистент не должен по умолчанию присылать полный replacement существующего production header, если пользователь этого явно не просил.
- По умолчанию пользователь должен иметь возможность:
  - скопировать новый skeleton-файл целиком;
  - скопировать нужные changed declaration blocks для существующего `.hpp`;
  - и затем самостоятельно дописать только тела методов.
- Тестовые файлы в такой постановке по-прежнему должны даваться целиком.

## 6. Правила приемки задач

### 6.1. Обязательные проверки при приемке
- При приемке реализации ассистент обязан отдельно проверить:
  - соответствие задаче;
  - соответствие плану;
  - соответствие правилам проектирования из `plan_rules.md`;
  - соответствие релевантным требованиям стандарта в пределах доступных источников;
  - отсутствие новых незафиксированных deviations;
  - расширяемость архитектуры;
  - качество тестового покрытия;
  - что временные ограничения локализованы и имеют follow-up task, если такие ограничения действительно неизбежны.

### 6.2. Архитектурная проверка при приемке
- При приемке ассистент обязан проверить, что реализация не закрепляет MVP-ограничения как архитектурные ограничения.
- В частности:
  - `PixelFormat` и `VideoScanMode` должны оставаться независимыми осями модели;
  - progressive-only поведение должно оставаться локализованным ограничением;
  - assumptions вида `marker == end of frame`, `timestamp group == frame`, `F is always zero` не должны появляться вне специально выделенных policy / mode-aware точек.
- Ассистент обязан отдельно проверять архитектуру на расширяемость при приемке каждой задачи.

### 6.3. Стандарты при приемке
- Приемка каждой задачи должна выполняться с учетом стандартного контекста ST 2110 / RP 2110, а не только по локальной логике кода или тестов.
- Перед приемкой реализации ассистент обязан реально прочитать:
  - все обязательные project control `.md` files в их текущих версиях;
  - все доступные в Project стандарты ST 2110 / RP 2110 в PDF.

- Приемка не может считаться standards-aware, если ассистент:
  - прочитал только один “наиболее релевантный” PDF;
  - не прочитал один из загруженных стандартных PDF;
  - опирался только на старое знание, цитаты из памяти или прежние обсуждения без повторного чтения актуальных файлов.

- Если нужные `.md` или PDF-файлы отсутствуют, недоступны или доступны только частично, ассистент обязан остановить приемку и запросить у пользователя конкретно недостающие файлы / фрагменты.
- Без актуального проектного и стандартного контекста ассистент не должен:
  - принимать задачу;
  - утверждать, что реализация соответствует стандарту;
  - закрывать deviation;
  - считать задачу полностью завершенной;
  - предлагать следующий шаг как продолжение уже принятой реализации.

- Ассистент не должен писать, что “изучил правила / работы / стандарты” перед приемкой, если не были реально прочитаны обязательные `.md` и все доступные стандартные PDF.

### 6.4. Расхождения, найденные при приемке
- Если во время приемки обнаружено расхождение со стандартом, правилами или целевой архитектурой, приоритет — исправить его сейчас же в рамках текущей задачи.
- Такая задача не считается принятой, пока расхождение не исправлено или пока пользователь явно не решит зафиксировать его как допустимое временное ограничение.
- Откладывать исправление допустимо только если:
  - исправление объективно выходит за границы текущей маленькой задачи;
  - или требует отдельного большого изменения;
  - или пользователь явно решает вынести его в отдельную задачу.
- В этом случае ассистент обязан:
  - зафиксировать расхождение в `Spec notes / deviations`;
  - сформулировать отдельную follow-up задачу;
  - явно указать, почему исправление не сделано немедленно.

- Исключение: если обнаруженное ограничение уже известно и уже покрыто существующей follow-up задачей в `plan.md`, а текущая задача не вносит нового отклонения, ассистент не должен дублировать его новым пунктом в `Spec notes / deviations`.
- В такой ситуации ассистент должен:
  - явно указать, что ограничение уже покрыто существующей задачей;
  - не создавать дублирующий deviation;
  - и проверять текущую задачу с учетом этого уже известного backlog item.

- Серьезное расхождение, найденное только на приемке, означает, что при формулировании задачи ассистент не учел важное требование стандарта, правил или архитектуры. Это считается грубой ошибкой ассистента и должно приводить к немедленной корректировке текущей задачи / критериев приемки.

## 7. Правила проектирования

### 7.1. Общая расширяемость
- Код должен быть написан в **расширяемом виде**.
- При добавлении новых форматов / режимов / backend’ов / типов потоков в типичных местах должно быть достаточно:
  - добавить новый enum/value;
  - дописать `case`/adapter/mapper;
  - добавить тесты;
  - а не переписывать существующую реализацию целиком.

### 7.2. Что нельзя жестко зашивать
- Нельзя жестко зашивать архитектуру только под:
  - один pixel format;
  - только video без возможности добавить audio;
  - только socket backend;
  - только консольный pipeline без дальнейшей OBS-интеграции.
- Нельзя жестко зашивать video pipeline только под progressive semantics.

### 7.3. Обязательные независимые оси video architecture
- Архитектурно должны быть разделены:
  - pixel/storage format;
  - scan mode (`Progressive | Interlaced | PsF`);
  - mode-dependent completion semantics (marker/timestamp/end-of-unit logic).
- Даже если MVP реализует только `Progressive`, код и API должны позволять добавить `Interlaced` и `PsF` через локальное расширение policy/state/config, а не через переписывание depacketizer/assembly pipeline.

### 7.4. MVP-ограничения не должны становиться архитектурными ограничениями
- **MVP может быть ограничен только в реализации, но не в архитектуре.**
- Любое известное или ожидаемо расширяемое измерение стандарта / runtime-модели / transport-модели / media-модели должно быть представлено явно, а не захардкожено в произвольном месте реализации.
- Запрещено закреплять текущий MVP-scope как неявное архитектурное ограничение через:
  - magic constants;
  - неименованные literals;
  - ad hoc `if` / `switch` без named boundary;
  - вычисляемые значения, записанные как фиксированные числа;
  - отсутствие поля / enum / helper / adapter / validation boundary для уже известной оси расширения.
- Допустимо временно поддерживать только часть вариантов стандарта, но только если это оформлено как:
  - explicit modeled representation;
  - explicit config/runtime/signaling axis, если значение влияет на behavior;
  - explicit validation/projection/support boundary;
  - named helper / named constant / named policy;
  - локализованный `Unsupported` / `InvalidValue` branch для пока не реализованных вариантов;
  - follow-up task, если это ограничение должно быть снято позже.
- Hardcoded значения допустимы только если они являются:
  - wire-format константами стандарта, оформленными как named constants;
  - локализованными current-support limits в явно названной support boundary;
  - тестовыми значениями внутри tests.
- Если значение может быть вычислено из signaled/runtime параметров, оно должно вычисляться через helper, а не записываться как фиксированная константа.
- Пример: audio `samples_per_packet` должен выводиться из `sampling_rate_hz` и `packet_time_us`, а не задаваться как `48`, даже если текущий Level A baseline дает именно `48`.

### 7.5. Стандартные и архитектурные оси, которые должны быть представлены явно
- Правило explicit modeled representation / boundary / dispatch / future branches относится ко всем известным расширяемым осям проекта, а не только к уже перечисленным примерам.
- В частности, явно должны моделироваться и не должны быть захардкожены:
  - media kind (`video` / `audio` / future media types);
  - backend kind (`socket` / `mtl` / future backend types);
  - pixel/storage format;
  - scan mode (`Progressive | Interlaced | PsF`);
  - packing mode (`GPM | BPM`);
  - RTP payload type admission;
  - clock / timestamp / timing / playout policy;
  - packet size / MAXUDP policy;
  - SDP/raw signaling scope and mapping boundaries;
  - video media-description properties;
  - audio sampling rate / packet time / samples-per-packet derivation;
  - audio conformance level / current runtime support boundary;
  - audio channel count / channel order / channel mapping;
  - receiver capability / support policy.
- Этот список не является закрытым. Если новая задача касается параметра, который может меняться по стандарту, SDP, runtime config, backend capabilities или future support, он должен рассматриваться как архитектурная ось или derived value до доказательства обратного.

### 7.6. Поздние реализации для уже заложенных веток
- Реализационные задачи для уже архитектурно заложенных веток **могут оставаться в `Medium` / later phases**.
- Это не нарушает MVP-принцип, если сама ось, boundary и dispatch уже заведены в MVP.
- Поздние фазы должны по возможности сводиться к:
  - заполнить уже существующий branch;
  - расширить enum/value coverage;
  - добавить adapter/mapper case;
  - добавить tests;
  - а не к:
    - впервые протащить ось через architecture;
    - переписать существующий pipeline;
    - заменить hardcoded assumption на полноценную модель.

### 7.7. Временные упрощения
- Все временные упрощения должны быть:
  - явно зафиксированы в плане;
  - локализованы;
  - оформлены как named support boundary / policy / adapter / helper;
  - сопровождаться отдельной задачей на устранение, если ограничение должно быть снято.
- Временное упрощение не должно маскироваться как окончательное архитектурное решение.
- Если во время задачи обнаружено, что значение или behavior были захардкожены вместо modeled/derived/boundary-based решения, текущая задача не считается принятой до исправления или явного вынесения ограничения в `Spec notes / deviations` и backlog.

### 7.8. Работа с отклонениями от стандартов
- **MVP не должен сознательно накапливать расхождения со стандартом**, если их можно избежать без взрывного роста сложности.
- Если найдено расхождение со стандартом, приоритет — исправить его в текущей задаче.
- Если немедленное исправление невозможно или явно вынесено пользователем в отдельную задачу, расхождение должно быть сразу занесено в:
  - `Spec notes / deviations`;
  - отдельную задачу на исправление или доведение до полного соответствия.

## 8. Цели этапов

- **MVP**: минимально полноценный прием ST 2110 для видео и аудио на Linux, с двумя backend’ами, базовой OBS-интеграцией и готовностью к ручному end-to-end тестированию.
- **Medium**: расширение форматов, улучшение устойчивости, более полная обработка edge-cases, улучшение UX/наблюдаемости, подготовка к полноценному тестированию.
- **Plugin**: доведение OBS-плагина до удобного и стабильного состояния для повседневного использования.
- **Tests**: систематическое полноценное тестирование, фиксация проблемных мест, regression-набор.
- **Hardening**: исправление узких мест, performance, recovery policy, дополнительная корректность.
- **Windows**: опциональный перенос только собственного socket backend, без MTL.

## 9. Референсы

- SMPTE ST 2110-10:2022 (system timing, definitions, common RTP/SDP/clock signaling requirements).
- SMPTE ST 2110-20:2022 (uncompressed video).
- SMPTE ST 2110-21:2022 (traffic shaping and delivery timing for video).
- SMPTE ST 2110-30:2025 (PCM digital audio).
- SMPTE RP 2110-25:2023 (measurement practices).
- RTP: RFC 3550 (структура заголовка, seq/timestamp/marker).
- Video over RTP: RFC 4175 (концепция packetization по строкам/фрагментам).
- Wireshark dissector ST2110-20 (для сверки полей SRD/ExtSeq/marker).
- Intel MTL (Media Transport Library) docs + st_pipeline_api (для MTL backend).
